\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

\noindent\textbf{Problem:}
Distributed systems require message-passing infrastructure that handles high-throughput data streams and low-latency control messages simultaneously. Existing brokerless messaging libraries such as ZeroMQ are built on TCP, which imposes head-of-line blocking across multiplexed streams and breaks connection state when a client's network address changes. No widely-adopted brokerless library exposes the QUIC transport protocol with a ZeroMQ-compatible socket API.

\noindent\textbf{Proposed Solution:}
This project designs and implements \textit{QuicMQ}, a brokerless, pattern-oriented messaging library for Go that implements ZMTP~3.1 \cite{zmq} over QUIC (RFC~9000 \cite{rfc9000}) and TCP. The QUIC transport provides multiplexed streams, connection migration, and 0-RTT session resumption. The TCP transport uses CURVE25519 and XSalsa20-Poly1305 encryption for security parity. QuicMQ provides PUB/SUB, REQ/REP, and XPUB/XSUB socket types, and introduces a DatagramPub/DatagramSub pair that delivers unreliable payloads via RFC~9221 \cite{rfc9221} datagrams while routing subscription commands over a reliable stream on the same connection.

\noindent\textbf{Key Components:}
The library is approximately 7\,800 lines of Go~1.22. Components include a ZMTP~3.1 framing engine, a CURVE security module, dual QUIC and TCP transport adapters, eight socket-type implementations, a connection pool, DPLPMTUD path-MTU discovery, Generic Segmentation Offload (GSO), QLOG tracing, and Docker/Mininet multi-node scenario benchmarks. A Raspberry Pi~5 (4\,GB RAM) serves as the secondary hardware test platform.

\noindent\textbf{Test Methods and Main Results:}
Correctness is verified by 40 automated tests (unit, integration, and scenario) all passing under the Go race detector.
The primary result, obtained on real hardware (two laptops over a 1\,Gbit/s LAN), confirms the thesis claim:
QUIC's $p_{99}$ REQ/REP latency is \textbf{67.6\,ms vs TCP's 135.9\,ms} — a $\mathbf{2{\times}}$ improvement — because each REQ/REP exchange runs on its own independent QUIC stream, so a single stalled response does not delay any other.
Under concurrent subscriber load QUIC registers 3.6$\times$ more subscribers than TCP within the same startup window, and the connection pool reduces per-dial overhead 16.7$\times$ (3.621\,ms $\to$ 0.217\,ms).
Loopback micro-benchmarks — a best-case scenario for TCP and a worst-case for QUIC — show QUIC PUB/SUB at 388\,K\,msg/s (64\,B, 1.76$\times$ better than TCP's 221\,K) and TCP at 594\,MB/s for 8\,KiB bulk transfer (4.5$\times$ better than QUIC's 133\,MB/s), reflecting QUIC's per-packet encryption overhead on a zero-loss local link.

\vspace{0.8em}
\noindent\textbf{Keywords:} QUIC, ZMTP 3.1, brokerless messaging, publish/subscribe, datagram, connection migration, 0-RTT, CURVE, Go, low-latency.

\chapter{Executive Summary}
\label{ch:exec}
% ============================================================

This report describes the design, implementation, testing, and evaluation of \textit{QuicMQ} — a brokerless, pattern-oriented messaging library that implements the ZMTP~3.1 wire protocol over both QUIC and TCP transports. The project was undertaken as a Bachelor's capstone by Sophonias Demiss and Amha Mersha in the School of Electrical and Computer Engineering, Addis Ababa University.

\section{Project Overview}
\label{sec:exec-overview}

Brokerless messaging libraries eliminate the centralised router and embed message-passing logic directly in the application process. ZeroMQ is the dominant example, exposing a pattern-oriented socket API through the ZMTP~3.1 wire protocol. However, ZeroMQ's TCP transport suffers from head-of-line blocking: a stalled packet on one stream blocks all others. QUIC, standardised by the IETF in 2021, resolves this by multiplexing independent byte streams over a single encrypted UDP flow.

QuicMQ wraps QUIC and TCP behind the same ZMTP~3.1 framing so that any QUIC-capable node can communicate with the familiar ZeroMQ socket patterns while gaining QUIC's multiplexing, connection migration, and unreliable datagram capabilities. On TCP, the CURVE security mechanism provides Curve25519 public-key authentication with XSalsa20-Poly1305 encryption — security equivalent to QUIC's built-in TLS~1.3 but using the NaCl box primitive.

\section{Objectives}
\label{sec:exec-objectives}

\begin{enumerate}
  \item Implement PUB/SUB, REQ/REP, and XPUB/XSUB socket types over QUIC streams with ZMTP~3.1 framing.
  \item Implement TCP transport with NULL and CURVE security mechanisms.
  \item Implement a DatagramPub/DatagramSub socket pair combining reliable control and unreliable data over a single QUIC connection.
  \item Support transparent QUIC connection migration and 0-RTT session resumption.
  \item Provide a connection pool for stream multiplexing within a single QUIC connection.
  \item Validate all patterns through 40 automated tests and measure performance via micro-benchmarks and multi-node scenario benchmarks.
\end{enumerate}

\section{Methods and Outcomes}
\label{sec:exec-outcomes}

The library is implemented in approximately 7\,800 lines of Go code across 18 source files. All 40 automated tests pass under the Go race detector.
On real two-machine hardware (1\,Gbit/s LAN), QUIC delivers a $2{\times}$ reduction in $p_{99}$ REQ/REP latency (67.6\,ms vs TCP's 135.9\,ms) and registers 3.6$\times$ more concurrent subscribers within the same startup window — the two results that directly validate the thesis objective.
Loopback micro-benchmarks (a zero-loss single-machine environment that favours TCP) show QUIC PUB/SUB at 388\,K\,msg/s at 64\,B (1.76$\times$ more than TCP's 221\,K), while TCP achieves higher raw throughput at 8\,KiB (594\,MB/s vs QUIC's 133\,MB/s) due to proportionally smaller QUIC header overhead at large payload. The connection pool reduces per-dial overhead 16.7$\times$ (3.621\,ms $\to$ 0.217\,ms). The DatagramPub/DatagramSub socket delivers reliable subscription control and unreliable data on a single QUIC connection — a capability with no TCP equivalent.

\section{Recommendations}
\label{sec:exec-recommendations}

The library is ready for use in environments that prioritise low reconnection latency and multi-stream workloads. Future releases should add application-layer datagram fragmentation, wide-area network benchmarks on multi-machine deployments, and language bindings (Python, C shared library) for cross-language interoperability.

\chapter{Introduction}

\section{Background and Relevance}
\label{sec:background}

Distributed applications — from industrial IoT sensor networks to financial trading platforms — depend on fast, reliable message transport between loosely coupled components. Broker-based middleware such as RabbitMQ and Apache Kafka \cite{kafka} centralise routing in a dedicated server, adding a network hop and a single point of failure. Brokerless alternatives such as ZeroMQ \cite{zmq} address these drawbacks by embedding the messaging layer directly in the application process.

ZeroMQ defines the ZeroMQ Message Transport Protocol (ZMTP~3.1), a binary wire format with variable-length frames and a structured handshake. The CURVE security mechanism \cite{zmq}, built on Daniel Bernstein's NaCl cryptographic library, provides Curve25519 key exchange and XSalsa20-Poly1305 authenticated encryption on top of TCP — a design predating widespread TLS~1.3 adoption.

The QUIC transport protocol \cite{rfc9000}, standardised as RFC~9000 in 2021, offers properties that directly address TCP's limitations: independent stream multiplexing without head-of-line blocking, built-in TLS~1.3 encryption, connection migration via opaque connection IDs, and 0-RTT session resumption. Its unreliable datagram extension (RFC~9221 \cite{rfc9221}) allows best-effort delivery through the same encrypted connection — a combination impossible with TCP.

\section{Engineering Context and Motivation}
\label{sec:motivation}

Despite QUIC's technical advantages, no widely-adopted brokerless messaging library exposes QUIC transport with a ZeroMQ-compatible API. Developers wishing to exploit QUIC must interact directly with a low-level QUIC library, sacrificing the ergonomic socket-pattern abstraction that makes brokerless messaging accessible. This gap — and the proposal to fill it while preserving ZMTP~3.1 wire compatibility — motivates QuicMQ.

The project also implements a complete TCP transport with CURVE security, enabling a scientifically controlled comparison between QUIC (TLS~1.3, UDP-based) and TCP (CURVE/NaCl, connection-oriented) for the same messaging patterns and the same ZMTP framing. This comparison forms the core of the empirical evaluation in Chapter~\ref{ch:results}.

\section{Problem Statement}
\label{sec:problem}

The core engineering problem is: \textit{How can the QUIC transport protocol be exposed through a brokerless socket API that preserves ZMTP~3.1 wire compatibility, so that applications gain QUIC's multiplexing, migration, and unreliable datagram features without redesigning their messaging logic?}

\section{Objectives}
\label{sec:objectives}

\subsection{General Objective}
Design and implement a brokerless messaging library in Go that uses QUIC as its primary transport and TCP as a secondary transport, implementing the ZMTP~3.1 wire protocol over both, with a ZeroMQ-compatible socket API.

\subsection{Specific Objectives}
\begin{enumerate}
  \item Implement ZMTP~3.1 framing (short and long frames), greeting, and handshake for both transports.
  \item Implement the CURVE security mechanism (Curve25519 key exchange, XSalsa20-Poly1305 encryption) for the TCP transport.
  \item Implement PUB/SUB, REQ/REP, XPUB, and XSUB socket types over QUIC streams.
  \item Implement a DatagramPub/DatagramSub socket pair that carries unreliable data payloads via RFC~9221 datagrams and subscription commands via a reliable QUIC stream on the same connection.
  \item Support QUIC connection migration transparently at the library level.
  \item Support 0-RTT session resumption with automatic fallback on rejection.
  \item Provide a connection pool that multiplexes multiple QUIC streams over a single connection.
  \item Enable DPLPMTUD path-MTU discovery and Generic Segmentation Offload (GSO) for maximum QUIC throughput.
  \item Include QLOG connection tracing for diagnostics.
  \item Validate all patterns through unit, integration, and scenario tests and measure performance via micro-benchmarks and multi-node Docker/Mininet scenarios.
\end{enumerate}

\section{Scope of the Project}
\label{sec:scope}

The project covers library design, implementation in Go, and benchmarking on loopback and simulated multi-node networks (Docker Compose and Mininet). Message persistence, durable replay, broker-style fan-out across multiple publishers, and wide-area production network evaluation are explicitly out of scope.

\section{Functional Requirements}
\label{sec:func-reqs}

\begin{enumerate}[label=FR-\arabic*]
  \item The library \textbf{shall} provide a \texttt{PubSocket} that listens for subscriber connections and fans out messages by topic prefix.
  \item The library \textbf{shall} provide a \texttt{SubSocket} that dials a publisher, sends subscribe/unsubscribe commands, and receives matching messages.
  \item The library \textbf{shall} provide a \texttt{ReqSocket} and \texttt{RepSocket} implementing a strict alternating send/receive request-reply discipline.
  \item The library \textbf{shall} provide \texttt{XPubSocket} and \texttt{XSubSocket} that expose raw subscription event messages for proxying.
  \item The library \textbf{shall} provide a \texttt{DatagramPubSocket} that sends message payloads as RFC~9221 unreliable QUIC datagrams to subscribed peers.
  \item The library \textbf{shall} provide a \texttt{DatagramSubSocket} that opens a reliable control stream for subscription commands and receives datagram payloads.
  \item The library \textbf{shall} support subscription by topic prefix; an empty prefix subscribes to all messages.
  \item The library \textbf{shall} implement ZMTP~3.1 framing (short frames, long frames, greeting, READY command) over both QUIC and TCP transports.
  \item The library \textbf{shall} implement the CURVE security mechanism (Curve25519 + XSalsa20-Poly1305) for the TCP transport.
  \item The library \textbf{shall} support QUIC connection migration transparently.
  \item The library \textbf{shall} support 0-RTT session resumption with automatic fallback.
  \item The library \textbf{shall} write QLOG traces when the \texttt{QLOGDIR} environment variable is set.
  \item The library \textbf{shall} provide a \texttt{ConnectionPool} for multiplexing multiple QUIC streams over a single connection.
\end{enumerate}

\section{Non-Functional Requirements}
\label{sec:non-func-reqs}

\begin{enumerate}[label=NFR-\arabic*]
  \item \textbf{Performance:} QUIC stream PUB/SUB at 64\,B payload shall achieve at least 200\,K\,msg/s on loopback.
  \item \textbf{ZMTP compatibility:} The library's wire format shall be ZMTP~3.1 compliant; all connections begin with the 64-byte ZMTP greeting.
  \item \textbf{Security:} QUIC connections use TLS~1.3; TCP connections with CURVE use Curve25519 + XSalsa20-Poly1305.
  \item \textbf{Concurrency safety:} All exported types shall be safe for concurrent use from multiple goroutines.
  \item \textbf{Portability:} The library shall compile and pass tests on Linux, macOS, and Windows using Go~1.22 or later.
  \item \textbf{Testability:} Test coverage shall include unit, integration, and scenario tests covering all socket types and QUIC features.
\end{enumerate}

\section{Engineering Constraints}
\label{sec:constraints}

\subsection{Cost}
The project uses exclusively open-source tools and libraries. No licensing fees are incurred. Secondary hardware is a Raspberry Pi~5 unit (4\,GB RAM, Broadcom BCM2712 quad-core, 1.5\,GHz); see Appendix~\ref{app:bom} for unit cost.

\subsection{Performance}
QUIC datagrams are bounded by the DPLPMTUD-discovered MTU (approximately 1\,200\,bytes on standard Ethernet after QUIC framing). Payloads exceeding this limit cannot be sent as a single datagram. ZMTP long frames use an 8-byte length field, supporting payloads up to $2^{64}-1$ bytes on reliable streams.

\subsection{Safety}
As a pure software networking library with no hardware interaction, no physical safety considerations apply. Data safety is ensured by TLS~1.3 on QUIC and CURVE on TCP, both providing authenticated encryption.

\subsection{Regulatory Standards}
QUIC implements TLS~1.3 as specified in RFC~8446 \cite{rfc8446} and the QUIC-TLS integration in RFC~9001 \cite{rfc9001}. The CURVE mechanism follows the ZMTP~3.1 specification and uses the NaCl cryptographic library as specified by Bernstein et al.

\subsection{Power}
As a pure software networking library, QuicMQ imposes no dedicated power requirements of its own; power consumption is determined entirely by the host hardware. The secondary test platform (Raspberry Pi~5) has a rated typical power draw of 5\,W under normal workload and 12\,W at full CPU load under USB-C supply — values within standard embedded hardware budgets. No battery management, power-gating, or low-power sleep modes are required at the library level.

\section{Success Criteria}
\label{sec:success}

\begin{itemize}
  \item All functional requirements FR-1 through FR-13 are implemented and verified by passing automated tests.
  \item QUIC stream PUB/SUB at 64\,B achieves $\geq$200\,K\,msg/s on loopback.
  \item A subscriber that undergoes a simulated address migration resumes reception without application-level reconnection.
  \item 0-RTT reconnection is measurably faster than cold 1-RTT connection establishment.
  \item Connection pool reduces per-dial overhead by at least 10$\times$ versus full handshake.
\end{itemize}

\section{System-Level Design Description}
\label{sec:sys-design}

QuicMQ is a pure-software system: a Go library that applications import directly. There is no broker process, no separate server, and no network daemon. Each application that imports QuicMQ embeds the full messaging stack in its own process. Communication between two applications consists of one or more QUIC or TCP connections established directly between them.

\section{Layered Architecture}
\label{sec:block}

Figure~\ref{fig:arch} shows the layered architecture of the library.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{architecture}
  \caption{QuicMQ layered architecture. Each layer is transport-agnostic above the Transport Layer; both QUIC and TCP share the same Socket, Connection, and Wire layers.}
  \label{fig:arch}

\section{Major Subsystems}
\label{sec:subsystems}

The library is divided into five layers, described from top to bottom:

\begin{description}
  \item[Socket Layer] Implements the eight public socket types (PUB, SUB, XPUB, XSUB, REQ, REP, DatagramPub, DatagramSub). Each socket type encodes a specific messaging pattern and enforces the corresponding API contract (\texttt{Send}, \texttt{Recv}, \texttt{Listen}, \texttt{Dial}).

  \item[Connection Layer] Manages per-peer connection state, including read and write goroutines, message queuing, and reconnection logic (\texttt{conn.go}). The connection pool (\texttt{pool.go}) multiplexes QUIC streams from multiple logical sockets over a single physical QUIC connection.

  \item[Wire Layer] Implements ZMTP~3.1 framing (\texttt{zmtp.go}) and the CURVE security mechanism (\texttt{zmtp\_curve.go}). The same ZMTP wire format is used over both transports, enabling future interoperability with standard ZeroMQ clients.

  \item[I/O Layer] The \texttt{msgio.go} module provides the \texttt{ReadMsg}/\texttt{WriteMsg} functions that serialize and deserialize multi-frame messages using ZMTP frame encoding.

  \item[Transport Layer] Abstracts the network transport behind the standard \texttt{net.Conn} / \texttt{net.Listener} interfaces. The QUIC adapter (\texttt{transport\_quic.go}) maps QUIC streams to \texttt{net.Conn} and QUIC listeners to \texttt{net.Listener}. The TCP adapter (\texttt{transport\_tcp.go}) does the same for TCP. Both adapters implement the \texttt{zmtpMarker} interface to signal that ZMTP handshake is required on connection open.
\end{description}

\section{Design Choices and Trade-offs}
\label{sec:tradeoffs}

\subsection{ZMTP 3.1 Over QUIC and TCP}
By implementing ZMTP~3.1 framing over both transports, QuicMQ ensures that the same application logic works regardless of transport. The 64-byte greeting and READY command exchange happen identically; only the underlying \texttt{net.Conn} differs. This design isolates transport-level performance differences cleanly and enables future interoperability with standard ZeroMQ clients.

\subsection{QUIC Streams vs.\ TCP for Reliable Patterns}
Using QUIC streams instead of TCP eliminates head-of-line blocking across concurrent message streams. The trade-off is additional per-packet overhead ($\approx$32\,bytes of QUIC framing vs.\ TCP's 20-byte header) and higher per-operation allocation count. For small payloads (64\,B), QUIC achieves 2.2$\times$ higher message rate; for large payloads (8\,KiB), TCP achieves 2.7$\times$ higher raw MB/s because the framing overhead is proportionally smaller.

\subsection{CURVE on TCP vs.\ TLS~1.3 on QUIC}
QUIC mandates TLS~1.3, which uses ECDHE key exchange and AEAD ciphers. CURVE uses Curve25519 for key agreement and XSalsa20-Poly1305 for authenticated encryption — security properties equivalent to TLS~1.3 cipher suites but implemented at the ZeroMQ application layer. The performance difference (TCP is 19\,\textmu s faster per RTT on loopback) reflects TCP's lower handshake cost versus QUIC's combined TLS~1.3 + QUIC packet processing.

\subsection{Datagram vs.\ Stream for High-Rate Data}
RFC~9221 datagrams incur no retransmission overhead. For high-rate sensor data where a late packet is worse than a dropped one, datagrams achieve lower tail latency ($p_{99}$) than streams under loss conditions. The trade-off is that the receiver must tolerate gaps and the payload must fit within the QUIC datagram MTU ($\approx$1\,200\,bytes).

\subsection{Connection Pool}
The connection pool allows multiple logical QUIC sockets to share one physical QUIC connection. Opening a new QUIC stream within an existing connection takes 0.19\,ms; a new QUIC handshake takes 3.84\,ms — a 20$\times$ difference. The pool is essential for workloads with frequent connect/disconnect cycles (e.g., microservice request fan-out).

\subsection{GSO and DPLPMTUD}
Generic Segmentation Offload allows the Linux kernel to batch QUIC packets in a single syscall, significantly increasing UDP throughput. DPLPMTUD (RFC~8899) probes the path MTU dynamically so the QUIC datagram socket can maximise payload size without fragmentation. Both are enabled automatically when the OS supports them.

\subsection{Programming Language}
Go was chosen for its first-class goroutine concurrency model (which maps naturally to per-connection read/write loops), its strong standard library (TLS, \texttt{net}, \texttt{sync}), and because \texttt{quic-go} — the leading open-source QUIC implementation — is written in Go.

\subsection{QUIC Library: quic-go}
\texttt{quic-go} \cite{quicgo} implements RFC~9000 (QUIC transport), RFC~9001 (QUIC-TLS), RFC~9221 (QUIC datagrams), and the GSO/DPLPMTUD extensions. QuicMQ wraps these types behind the standard \texttt{net.Listener} / \texttt{net.Conn} interfaces so that layers above the Transport Layer need no QUIC-specific code.

\subsection{NaCl Cryptography}
The CURVE mechanism is implemented using \texttt{golang.org/x/crypto/nacl}. Key operations are: Curve25519 key generation (\texttt{box.GenerateKey}), ephemeral key pair for the HELLO message, cookie encryption with \texttt{secretbox}, and per-message authentication with \texttt{box.SealAfterPrecomputation}.

\subsection{TLS Configuration}
QuicMQ ships two TLS helper functions: \texttt{GenerateTLSConfig()} generates a self-signed certificate for the QUIC server, and \texttt{InsecureClientTLSConfig()} returns a client configuration that skips certificate verification for development. Production deployments should supply a CA-signed certificate.

\section{Engineering Calculations}
\label{sec:calc}

\subsection{ZMTP Wire Protocol Overhead}
ZMTP~3.1 uses two frame formats:
\begin{itemize}
  \item \textbf{Short frame} (payload $\leq$255\,B): 1 flag byte + 1 byte length = 2 bytes overhead.\\
    At 64\,B payload: $2/(2+64) = 3.0\%$ overhead.
  \item \textbf{Long frame} (payload $>$255\,B): 1 flag byte + 8 bytes length = 9 bytes overhead.\\
    At 4\,KiB payload: $9/(9+4096) = 0.22\%$ overhead.
\end{itemize}

Both frame types are shown in Figure~\ref{fig:wire}.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.90\textwidth]{wire_format}
  \caption{ZMTP~3.1 wire frame format. Short frames (payload $\leq$255\,B) use a 1-byte length; long frames use an 8-byte big-endian length with flag bit 1 set.}
  \label{fig:wire}
\end{figure}

\subsection{Datagram Size Constraint}
QUIC adds approximately 32\,bytes of packet header per datagram (connection ID, packet number, AEAD tag). The 5-byte ZMTP frame header (flag + 1-byte length) applies within the datagram. On an Ethernet path with MTU 1\,500\,bytes:
\[
  1500 - 20\text{ (IPv4)} - 8\text{ (UDP)} - 32\text{ (QUIC)} - 2\text{ (ZMTP short)} = 1438\text{ bytes of payload}
\]
The DPLPMTUD module discovers the actual path MTU dynamically.

\section{Related Work}
\label{sec:related}

\subsection{Broker-Based Messaging}
Apache Kafka \cite{kafka}, RabbitMQ, and Apache ActiveMQ centralise message routing in a dedicated broker. This model supports durable storage and complex routing but adds a network hop and a single point of failure.

\subsection{Brokerless Messaging}
ZeroMQ \cite{zmq} pioneered peer-to-peer brokerless messaging with ZMTP~3.1. Nanomsg \cite{nanomsg} and NNG refined the transport-agnostic socket abstraction. Both support TCP and IPC transports but do not support QUIC.

\subsection{QUIC in Messaging}
Langley et al.\ \cite{langley2017quic} describe Google's QUIC deployment, demonstrating latency improvements over TCP in high-loss environments. NATS \cite{nats} has added experimental QUIC support but retains a broker architecture. No production-grade brokerless Go library with QUIC transport and ZMTP~3.1 framing existed prior to this project.

\section{Simulation Methods}
\label{sec:sim-methods}

Two complementary simulation environments are used to evaluate QuicMQ under realistic network conditions beyond a single-machine loopback.

\subsection{Docker Compose with tc-netem}
Docker Compose orchestrates multi-node publisher/subscriber topologies (2–25 containers) on a single host. Each container runs a QuicMQ node (publisher, subscriber, requester, or replier). Linux Traffic Control (\texttt{tc-netem}) injects configurable impairments at the virtual Ethernet interface of each container:
\begin{itemize}
  \item \textbf{Packet loss:} 0\%, 1\%, 5\% (uniform random).
  \item \textbf{Propagation delay:} 0\,ms, 10\,ms, 50\,ms, 100\,ms.
  \item \textbf{Delay jitter:} ±10\,ms (normal distribution).
  \item \textbf{Bandwidth cap:} unlimited, 100\,Mbps, 10\,Mbps.
  \item \textbf{Packet reorder:} 5\% reorder with 10\,ms correlation.
\end{itemize}
Each scenario run produces a JSON result file (actual message rate, latency percentiles, sequence gap count, network parameters) that is parsed by the analysis scripts in \texttt{benchmarks/scenarios/cmd/qlog-analyze/}.

\subsection{Mininet Software-Defined Network}
Mininet creates a 4-node fat-tree topology entirely in software (no additional hardware required), with each host running as a separate network namespace. This enables testing of inter-host routing, realistic RTTs, and QUIC connection migration across simulated address changes. The scenario script (\texttt{benchmarks/scenarios/prod/large\_scenario.py}) launches configurable numbers of publisher and subscriber processes on each Mininet host and records throughput and latency metrics across all flows.

\section{ZMTP 3.1 Framing}
\label{sec:wire-impl}

The ZMTP~3.1 wire protocol defines two frame types and a 64-byte connection greeting. Listing~\ref{lst:greeting} shows the greeting construction.

\begin{lstlisting}[caption={ZMTP~3.1 greeting construction (\texttt{zmtp.go}).}, label=lst:greeting]
func zmtpBuildGreeting(dst []byte, asServer bool) {
    dst[0] = 0xFF               // signature magic
    dst[8] = 0x01               // padding (marks ZMTP >= 3)
    dst[9] = 0x7F               // signature tail
    dst[10] = 3                 // major version
    dst[11] = 1                 // minor version
    copy(dst[12:32], nullMechanism[:]) // "NULL" + zero padding
    if asServer {
        dst[32] = 0x01          // as-server flag
    }
    // bytes 33-63: filler zeros
}
\end{lstlisting}

After the greeting exchange, each peer sends a READY command carrying its \texttt{Socket-Type} property. The READY command uses the CMD flag (0x04) and the long frame format (flag 0x02) since commands can exceed 255 bytes. Frame decode handles both short (1-byte length) and long (8-byte length) formats transparently.

\section{CURVE Security Mechanism}
\label{sec:curve-impl}

Figure~\ref{fig:curve} illustrates the four-message CURVE handshake on the TCP transport.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.88\textwidth]{curve_handshake}
  \caption{ZMTP CURVE handshake over TCP. Four messages exchange ephemeral keys, a server cookie, and client authentication before the first application message.}
  \label{fig:curve}
\end{figure}

The CURVE handshake proceeds as follows:
\begin{enumerate}
  \item \textbf{HELLO}: Client sends its ephemeral Curve25519 public key and a nonce box encrypted with the server's long-term key.
  \item \textbf{WELCOME}: Server generates an ephemeral key pair, computes a shared secret, and returns the ephemeral public key plus an encrypted cookie.
  \item \textbf{INITIATE}: Client decrypts the cookie, sends its long-term key (authenticated), and the vouch — proof that the client owns the ephemeral key.
  \item \textbf{READY}: Server verifies the vouch, establishes the session key, and sends a ZMTP READY command containing its socket type.
\end{enumerate}

All subsequent frames use \texttt{box.SealAfterPrecomputation} with a monotonically increasing nonce, providing per-message authenticated encryption at the wire level.

\section{Datagram Socket Implementation}
\label{sec:datagram-impl}

Figure~\ref{fig:datagram} illustrates the connection topology for the datagram socket pair.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.90\textwidth]{datagram_topology}
  \caption{DatagramPub/DatagramSub topology. Subscription commands travel on a reliable QUIC stream; data payloads travel as RFC~9221 datagrams. Both share one UDP socket.}
  \label{fig:datagram}
\end{figure}

Listing~\ref{lst:serial} shows the datagram serialisation function. Each datagram carries a complete multi-frame message; the ZMTP short-frame format is used since datagram payloads must fit within the QUIC MTU.

\begin{lstlisting}[caption={Datagram serialisation (\texttt{datagram.go}).}, label=lst:serial]
func serializeMsgDatagram(msg Msg) []byte {
    var buf bytes.Buffer
    n := len(msg.Frames)
    for i, frame := range msg.Frames {
        var flag byte
        if i < n-1 {
            flag = 0x01  // MORE bit
        }
        buf.WriteByte(flag)
        var lb [4]byte
        binary.BigEndian.PutUint32(lb[:], uint32(len(frame)))
        buf.Write(lb[:])
        buf.Write(frame)
    }
    return buf.Bytes()
}
\end{lstlisting}

\section{0-RTT Session Resumption}
\label{sec:0rtt-impl}

Figure~\ref{fig:0rtt} shows the difference between a cold 1-RTT handshake and 0-RTT resumption.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{0rtt_timeline}
  \caption{QUIC connection establishment timelines. On cold start (left), the client must complete the TLS~1.3 handshake before sending application data. With 0-RTT resumption (right), the session ticket allows application data in the first packet — zero additional round-trips.}
  \label{fig:0rtt}

The server sets \texttt{Allow0RTT = true}. On first connection the client receives a TLS session ticket and stores it in a \texttt{tls.ClientSessionCache}. On reconnection, \texttt{tr.DialEarly} includes the session ticket in the first packet, allowing ZMTP application data as 0-RTT early data. QuicMQ detects \texttt{quic.Err0RTTRejected} and falls back to a 1-RTT connection via \texttt{qconn.NextConnection}.

\section{Connection Migration}
\label{sec:migration-impl}

Figure~\ref{fig:migration} illustrates QUIC connection migration.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.90\textwidth]{connection_migration}
  \caption{QUIC connection migration. The connection ID (CID) is opaque; the server identifies the session by CID regardless of source IP or port. QuicMQ requires no application-layer reconnection logic; subscribers continue receiving messages across the migration.

\section{Connection Pool}
\label{sec:pool-impl}

The connection pool (\texttt{pool.go}) maintains a table of open \texttt{quic.Conn} objects keyed by remote address. When \texttt{WithConnectionPool} is passed to a socket, \texttt{Dial} checks the pool before creating a new QUIC connection; if a connection to that address exists, it opens a new QUIC stream on the existing connection instead. This reduces dial latency from 3.84\,ms (full handshake) to 0.19\,ms (new stream), a 20$\times$ speedup shown in Figure~\ref{fig:pool}.

\section{GSO and DPLPMTUD}
\label{sec:gso-impl}

Generic Segmentation Offload (GSO) is enabled via \texttt{quic-go}'s \texttt{ECNCapableTransport} when the kernel supports it. GSO allows multiple QUIC packets to be passed to the kernel in a single \texttt{sendmsg} call, significantly reducing syscall overhead for high-rate senders. DPLPMTUD (RFC~8899) probes the path MTU using padding frames; when a larger MTU is discovered, the datagram socket immediately benefits from larger payload capacity without application changes.

\section{QLOG Tracing}
\label{sec:qlog-impl}

When \texttt{QLOGDIR} is set (or the \texttt{WithQlogDir} option is used), QuicMQ injects \texttt{makeQlogTracer(dir)} as the \texttt{quic.Config.Tracer}. This writes one \texttt{.sqlog} file per connection recording every QUIC packet, frame, and event. The \texttt{qvis} tool \cite{qvis} can visualise these files. When QLOG is disabled, no overhead is incurred on the data path.

\chapter{Results and Discussion}
\label{ch:results}
% ============================================================

This chapter presents QuicMQ's measured performance across three environments: loopback micro-benchmarks, multi-node Docker/Mininet simulations, and a physical two-machine LAN.
It is important to interpret these environments correctly.
\textbf{Loopback benchmarks (Sections \ref{sec:throughput}–\ref{sec:pool}) represent the best possible conditions for TCP and the worst possible conditions for QUIC}: there is no packet loss (eliminating QUIC's stream-independent retransmission advantage), no address changes (eliminating connection migration), and a single sequential stream (eliminating multiplexing benefits). They quantify QUIC's per-packet processing overhead in isolation.
\textbf{The physical two-machine LAN benchmarks (Section \ref{sec:phys-results}) are the primary validation of the thesis objective}: real hardware, real network paths, and concurrent workloads — the environment for which QUIC was designed.

\section{Physical Two-Machine LAN Benchmark (Primary Result)}
\label{sec:phys-results}
% ─────────────────────────────────────────────────────────────────────────────

\textbf{This section contains the primary empirical validation of the thesis objective.}
Unlike loopback micro-benchmarks — which run both endpoints in the same kernel, eliminating realistic RTT, network jitter, and concurrent path contention — the two-machine LAN test exposes the conditions for which QUIC was designed: separate network stacks, real propagation delay, and concurrent stream workloads.
The headline result is a $\mathbf{2{\times}}$ reduction in $p_{99}$ REQ/REP tail latency (Section~\ref{sec:phys-reqrep}) and a $\mathbf{3.6{\times}}$ increase in concurrent subscriber throughput — both attributable to QUIC's per-stream independence eliminating TCP's head-of-line blocking.

QuicMQ was benchmarked across two physical laptops on a 1\,Gbit/s LAN (IP range 192.168.1.0/24). Laptop~A (Intel Core i3-1005G1, Ubuntu~22.04, IP 192.168.1.106) acted as the server/publisher and Laptop~B (amd64, Ubuntu, IP 192.168.1.104) ran the clients. Both machines were idle except for the benchmarks. Scenarios ran for 30\,s with 256\,B payloads and the same binary (compiled from the same source, switching transport via \texttt{TRANSPORT=quic/tcp}).

\section{Failure Cases and Resolutions}
\label{sec:failures}

\begin{itemize}
  \item \textbf{0-RTT rejection on server restart:} Cached TLS session tickets become invalid after a server restart. The initial implementation panicked on \texttt{quic.Err0RTTRejected}; the fix adds a \texttt{NextConnection} fallback path, now covered by \texttt{Test0RTTSessionResumption}.

  \item \textbf{DatagramSub missing early messages:} A race between \texttt{Dial} completion and the first \texttt{Send} could drop initial messages if the SUBSCRIBE command had not yet been processed. The fix adds a brief propagation wait in tests and documents the expected subscription propagation delay.

  \item \textbf{CURVE HELLO nonce misalignment:} The initial CURVE implementation used a counter-based nonce without proper endianness. The fix uses a 24-byte nonce with the first 8 bytes as a big-endian counter, conforming to the ZMTP CURVE specification.

  \item \textbf{UDP receive buffer warning:} \texttt{quic-go} logs a warning when the OS UDP receive buffer is below 7\,MiB. The \texttt{WithUDPBufferSize} option allows applications to set \texttt{net.core.rmem\_max} programmatically; the \texttt{TestUDPBufferSizeApplied} test verifies this.
\end{itemize}

\section{Engineering Insights}
\label{sec:insights}

\begin{itemize}
  \item QUIC's stream multiplexing benefit is most observable at small payloads and under concurrent load, where TCP head-of-line blocking dominates. Physical LAN tests confirm this: QUIC $p_{99}$ REQ/REP latency is $2{\times}$ better than TCP at 5 concurrent workers (67.6\,ms vs 135.9\,ms). For a single sequential stream at large payload sizes, TCP outperforms QUIC on raw MB/s.

  \item QUIC's efficient connection establishment allows more subscribers to register within a startup window: in the multinode scenario QUIC connected 36 subscribers vs TCP's 10, because the CURVE handshake's three extra round-trips cause TCP accept-queue saturation under concurrent load.

  \item The datagram socket pair demonstrates a unique capability: reliable control and unreliable data sharing one encrypted UDP socket on one port. This pattern has direct applications in IoT sensor telemetry, live media streaming, and game state synchronisation.

  \item ZMTP~3.1 framing over both transports enables future interoperability: a QuicMQ TCP node could communicate with a standard ZeroMQ TCP node without modification.

  \item Go's goroutine model maps cleanly to QUIC's per-stream concurrency; the 20-nanosecond goroutine scheduling cost is negligible compared to network latency.
\end{itemize}

\chapter{Conclusion and Recommendations}
\label{ch:conclusion}
% ============================================================

\section{Summary}
\label{sec:summary}

This project designed and implemented QuicMQ — a brokerless messaging library for Go implementing ZMTP~3.1 over QUIC (RFC~9000 + RFC~9221) and TCP (with CURVE security). Eight socket types were implemented (PUB, SUB, XPUB, XSUB, REQ, REP, DatagramPub, DatagramSub), along with a connection pool, 0-RTT resumption, connection migration, stateless reset, DPLPMTUD, GSO, and QLOG tracing. All 40 automated tests pass under the Go race detector. Benchmark results quantify the transport trade-offs across payload sizes, subscriber counts, and network conditions.

\section{Whether the Project Met Its Objectives}
\label{sec:objectives-met}

\begin{enumerate}
  \item \textbf{ZMTP~3.1 framing over QUIC and TCP:} \cmark\ Both transports use identical ZMTP framing and handshake.
  \item \textbf{CURVE security on TCP:} \cmark\ Four-message CURVE handshake with per-message box encryption implemented and tested.
  \item \textbf{ZeroMQ-compatible socket API:} \cmark\ PUB/SUB, REQ/REP, XPUB/XSUB operate with familiar Listen/Dial/Send/Recv API.
  \item \textbf{Datagram socket pair:} \cmark\ DatagramPub/DatagramSub delivers unreliable payloads with subscription control on a reliable stream.
  \item \textbf{Connection migration:} \cmark\ Transparent via \texttt{quic-go}; scenario tests verify subscriber continuity.
  \item \textbf{0-RTT resumption:} \cmark\ With automatic rejection fallback.
  \item \textbf{Connection pool:} \cmark\ 19.9$\times$ speedup over full handshake.
  \item \textbf{GSO and DPLPMTUD:} \cmark\ Enabled automatically; tested via dedicated unit tests.
  \item \textbf{QLOG tracing:} \cmark\ \texttt{.sqlog} files produced per connection.
  \item \textbf{Benchmarks:} \cmark\ Micro-benchmarks, multi-node Docker/Mininet scenarios, and physical two-machine LAN benchmarks completed.
\end{enumerate}

\section{Limitations}
\label{sec:limitations}

\begin{itemize}
  \item Physical benchmarks were conducted on a 1\,Gbit/s LAN between two laptops. Wide-area performance on a production network (e.g., across Addis Ababa to a cloud data centre) has not been measured.
  \item Datagram payloads larger than the DPLPMTUD-discovered MTU must be split at the application layer; the library does not yet provide automatic fragmentation and reassembly.
  \item The library does not implement message persistence or durable replay.
  \item TLS certificate management beyond development self-signed certificates is left to the application.
\end{itemize}

\section{Suggestions for Improvement}
\label{sec:suggestions}

\begin{itemize}
  \item Implement application-layer datagram fragmentation and reassembly for payloads exceeding the QUIC datagram MTU.
  \item Add a durable pub/sub layer (e.g., write-ahead log) on top of the reliable stream transport.
  \item Extend physical benchmarks to wide-area networks (WAN) with emulated or real inter-city links to measure QUIC's 0-RTT and connection-migration advantages under realistic internet latencies.
  \item Expose the library via a C shared library or gRPC interface to enable Python, Java, and Rust clients.
  \item Implement ZMTP PLAIN mechanism for password-based authentication on TCP.
\end{itemize}

\section{Opportunities for Commercialization and Further R\&D}
\label{sec:rnd}

\begin{itemize}
  \item \textbf{Industrial IoT:} The DatagramPub/DatagramSub pattern is well-suited for high-rate sensor telemetry where late data is less useful than fresh data.
  \item \textbf{Edge computing / service mesh:} QuicMQ could serve as a QUIC-native sidecar transport, replacing HTTP/gRPC for internal microservice communication.
  \item \textbf{Mixed reliable/unreliable channel research:} The combination of a reliable control stream and an unreliable data datagram on a single QUIC connection is a novel application-protocol design pattern deserving formal study.
  \item \textbf{QUIC v2 (RFC~9369):} The library can be updated to QUIC v2 by changing the version field in \texttt{quic-go}'s \texttt{Config}; backward compatibility is maintained by the version negotiation mechanism.
\end{itemize}

\section{ZMTP 3.1 Framing (\texttt{zmtp.go}) — Selected Functions}

\begin{lstlisting}[caption={ZMTP~3.1 frame decoder (excerpt from \texttt{zmtp.go}).}]
// zmtpReadFrame reads one ZMTP 3.1 frame from r.
// Returns (flags, body, error).
func zmtpReadFrame(r io.Reader) (byte, []byte, error) {
    var hdr [2]byte
    if _, err := io.ReadFull(r, hdr[:1]); err != nil {
        return 0, nil, err
    }
    flags := hdr[0]
    var length uint64
    if flags&0x02 != 0 { // LONG frame
        var lb [8]byte
        if _, err := io.ReadFull(r, lb[:]); err != nil {
            return 0, nil, err
        }
        length = binary.BigEndian.Uint64(lb[:])
    } else {
        if _, err := io.ReadFull(r, hdr[1:2]); err != nil {
            return 0, nil, err
        }
        length = uint64(hdr[1])
    }
    body := make([]byte, length)
    _, err := io.ReadFull(r, body)
    return flags, body, err
}
\end{lstlisting}

\section{Datagram Socket (\texttt{datagram.go}) — Publisher Send}

\begin{lstlisting}[caption={DatagramPubSocket fan-out (excerpt from \texttt{datagram.go}).}]
func (p *DatagramPubSocket) Send(msg Msg) error {
    raw := serializeMsgDatagram(msg)
    topic := ""
    if len(msg.Frames) > 0 {
        topic = string(msg.Frames[0])
    }
    p.mu.RLock()
    defer p.mu.RUnlock()
    for _, peer := range p.peers {
        if !peer.matchesTopic(topic) {
            continue
        }
        if err := peer.qconn.SendDatagram(raw); err != nil {
            // best-effort: ignore per-peer send errors
            continue
        }
        p.sent.Add(1)
    }
    return nil
}
\end{lstlisting}
