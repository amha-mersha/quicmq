\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

\noindent\textbf{Problem:}
Distributed systems require message-passing infrastructure that supports simultaneous high-throughput data streams and low-latency control messages. Existing brokerless messaging libraries — such as ZeroMQ — are built on TCP, which imposes head-of-line blocking across multiplexed streams and prevents transparent connection migration when a client's network address changes. No widely-adopted library exposes the QUIC transport protocol in a ZeroMQ-compatible socket API.

\noindent\textbf{Proposed Solution:}
This project designs and implements \textit{QuicMQ}, a brokerless, pattern-oriented messaging library for Go that uses QUIC (RFC 9000) as its transport. QuicMQ provides familiar socket patterns (PUB/SUB, REQ/REP, XPUB/XSUB) over multiplexed QUIC streams and introduces a novel datagram socket pair (DatagramPub/DatagramSub) that delivers unreliable data payloads via QUIC RFC 9221 datagrams while routing subscription commands over a reliable stream on the same connection.

\noindent\textbf{Key Components:}
The library is implemented entirely in Go~1.22 and depends on the \texttt{quic-go} library for RFC-compliant QUIC transport. Core software components include a transport abstraction layer, a binary wire-framing module, eight socket-type implementations, a connection pool, TLS key management helpers, and a QLOG tracing module. No custom hardware is required.

\noindent\textbf{Test Methods and Main Results:}
Correctness is verified through a suite of unit, integration, and scenario tests covering all socket types, connection migration, 0-RTT resumption, and datagram topic filtering. Performance is measured using Go's built-in benchmarking framework across publish/subscribe and request/reply workloads with 64\,B and 4\,KiB payloads. All functional tests pass; benchmark results are reported in Chapter~8.

\vspace{0.8em}
\noindent\textbf{Keywords:} QUIC, brokerless messaging, publish/subscribe, datagram, low-latency, connection migration, 0-RTT, Go.

\chapter{Executive Summary}
\label{ch:exec}
% ============================================================

This report describes the design, implementation, testing, and evaluation of \textit{QuicMQ} — a brokerless, pattern-oriented messaging library built on the QUIC transport protocol. The project was undertaken as a Bachelor's capstone by Sophonias Demiss and Amha Mersha in the School of Electrical and Computer Engineering, Addis Ababa University.

\section{Project Overview}
\label{sec:exec-overview}

Brokerless messaging libraries eliminate the centralised router and embed message-passing logic directly in the application process. ZeroMQ is the dominant example, but it relies on TCP — a protocol that suffers from head-of-line blocking when multiple streams share one connection, and that breaks when a client's IP address changes. QUIC, the transport protocol standardised by the IETF in 2021, solves both problems: it multiplexes independent byte streams over a single encrypted UDP flow and migrates connections across network address changes transparently.

QuicMQ wraps QUIC behind the same socket-pattern interface developers know from ZeroMQ: publish/subscribe, request/reply, and extended pub/sub. It also introduces a \textit{datagram socket pair} — a novel pattern that sends lossy sensor-style data as RFC~9221 QUIC datagrams while keeping subscription control on a reliable stream of the same connection.

\section{Objectives}
\label{sec:exec-objectives}

\begin{enumerate}
  \item Implement PUB/SUB, REQ/REP, and XPUB/XSUB socket types over QUIC streams with a ZeroMQ-compatible API.
  \item Implement a DatagramPub/DatagramSub socket pair that combines reliable control and unreliable data over a single QUIC connection.
  \item Support transparent connection migration and 0-RTT session resumption.
  \item Validate all patterns through automated tests and measure performance via micro-benchmarks.
\end{enumerate}

\section{Methods and Outcomes}
\label{sec:exec-outcomes}

The library is implemented in approximately 2\,000 lines of Go code, relying on \texttt{quic-go} as the sole third-party QUIC dependency. All eight socket types pass their functional test suites. Benchmark results show that QUIC stream-based PUB/SUB achieves throughput competitive with equivalent TCP-based messaging at small payload sizes, with reduced tail latency under concurrent-stream workloads where TCP's head-of-line blocking would otherwise dominate. The datagram socket pair achieves the lowest latency of any pattern, at the cost of best-effort delivery.

\section{Recommendations}
\label{sec:exec-recommendations}

The library is ready for use in environments that prioritise low reconnection latency and multi-stream workloads. Future releases should add wide-area network benchmarks, application-layer datagram fragmentation for payloads exceeding the QUIC datagram MTU, and language bindings beyond Go.

\chapter{Introduction}

\section{Background and Relevance}
\label{sec:background}

Distributed applications — from industrial IoT sensor networks to financial trading platforms — depend on fast, reliable message transport between loosely coupled components. The dominant approach for decades has been broker-based middleware such as RabbitMQ and Apache Kafka \cite{kafka}, which centralise routing, persistence, and fan-out logic in a dedicated server. While effective at scale, brokers introduce a single point of failure, add a network hop of latency, and require significant operational infrastructure.

Brokerless alternatives such as ZeroMQ \cite{zmq} address these drawbacks by embedding the messaging layer directly in the application process, eliminating the broker hop entirely. ZeroMQ's socket abstraction — publish/subscribe, request/reply, push/pull — has proven highly ergonomic and has influenced subsequent libraries including Nanomsg \cite{nanomsg} and NNG. However, all of these libraries build on TCP, which imposes head-of-line blocking when multiple message streams share one connection: a stalled or lost packet on one stream blocks all others until it is retransmitted.

The QUIC transport protocol \cite{rfc9000}, standardised by the IETF as RFC~9000 in 2021, removes this limitation. QUIC multiplexes independent byte streams over a single UDP flow; loss on one stream does not stall others. QUIC also provides built-in TLS~1.3 encryption, connection migration based on opaque connection IDs (so a mobile client survives a network handover), and 0-RTT session resumption that eliminates the handshake round-trip on reconnection. Its unreliable datagram extension (RFC~9221 \cite{rfc9221}) further allows best-effort datagrams through the same encrypted connection, enabling mixed reliable/unreliable delivery over a single UDP socket — a combination impossible with TCP.

\section{Engineering Context and Motivation}
\label{sec:motivation}

Despite QUIC's technical advantages, no widely-adopted brokerless messaging library exposes QUIC transport natively with a ZeroMQ-compatible API. Developers who wish to exploit QUIC's features must interact directly with a low-level QUIC library, sacrificing the ergonomic socket-pattern abstraction that makes brokerless messaging accessible. This gap motivates the construction of QuicMQ.

\section{Problem Statement}
\label{sec:problem}

The core engineering problem is: \textit{how can the QUIC transport protocol be exposed through a familiar brokerless socket API so that applications gain QUIC's multiplexing, migration, and unreliable datagram features without sacrificing the simplicity of pattern-oriented messaging?}

\section{Objectives}
\label{sec:objectives}

\subsection{General Objective}
Design and implement a brokerless messaging library in Go that uses QUIC as its sole transport, exposing a ZeroMQ-compatible socket API.

\subsection{Specific Objectives}
\begin{enumerate}
  \item Implement PUB/SUB, REQ/REP, XPUB, and XSUB socket types over QUIC streams.
  \item Implement a DatagramPub/DatagramSub socket pair that carries unreliable data payloads via RFC~9221 datagrams and subscription commands via a reliable QUIC stream on the same connection.
  \item Support QUIC connection migration transparently at the library level.
  \item Support 0-RTT session resumption with automatic fallback on rejection.
  \item Include QLOG connection tracing for diagnostics.
  \item Validate all patterns through unit, integration, and scenario tests.
  \item Measure throughput and latency via micro-benchmarks.
\end{enumerate}

\section{Scope of the Project}
\label{sec:scope}

The project covers library design, implementation in Go, and loopback-network benchmarking. Message persistence, durable replay, broker-style fan-out across multiple publishers, and wide-area network evaluation are explicitly out of scope. No custom hardware is required.

\section{Functional Requirements}
\label{sec:func-reqs}

\begin{enumerate}[label=FR-\arabic*]
  \item The library \textbf{shall} provide a \texttt{PubSocket} that listens for subscriber connections and fans out messages by topic prefix.
  \item The library \textbf{shall} provide a \texttt{SubSocket} that dials a publisher, sends subscribe/unsubscribe commands, and receives matching messages.
  \item The library \textbf{shall} provide a \texttt{ReqSocket} and \texttt{RepSocket} implementing a strict alternating send/receive request-reply discipline.
  \item The library \textbf{shall} provide \texttt{XPubSocket} and \texttt{XSubSocket} that expose raw subscription event messages, enabling proxying.
  \item The library \textbf{shall} provide a \texttt{DatagramPubSocket} that sends message payloads as RFC~9221 unreliable QUIC datagrams to subscribed peers.
  \item The library \textbf{shall} provide a \texttt{DatagramSubSocket} that dials a \texttt{DatagramPubSocket}, opens a reliable control stream for subscription commands, and receives datagram payloads.
  \item The library \textbf{shall} support subscription by topic prefix; an empty topic prefix shall subscribe to all messages.
  \item The library \textbf{shall} support QUIC connection migration transparently, without application-level reconnection logic.
  \item The library \textbf{shall} support 0-RTT session resumption and shall fall back to 1-RTT gracefully if the server rejects early data.
  \item The library \textbf{shall} write QLOG traces when the \texttt{QLOGDIR} environment variable is set.
\end{enumerate}

\section{Non-Functional Requirements}
\label{sec:non-func-reqs}

\begin{enumerate}[label=NFR-\arabic*]
  \item \textbf{Performance:} PUB/SUB throughput shall meet or exceed the performance of an equivalent loopback TCP socket at 64\,B payload size.
  \item \textbf{Reliability:} Stream-based socket types shall use QUIC's built-in reliable delivery; no message shall be silently dropped on a healthy connection.
  \item \textbf{Security:} Every connection shall use TLS~1.3; there shall be no plaintext transport mode for production use.
  \item \textbf{Concurrency safety:} All exported types shall be safe for concurrent use from multiple goroutines.
  \item \textbf{Portability:} The library shall compile and pass tests on Linux, macOS, and Windows using Go~1.22 or later.
  \item \textbf{Testability:} Test coverage shall include unit tests per socket type, integration tests verifying end-to-end message delivery, and scenario tests for connection migration and 0-RTT.
\end{enumerate}

\section{Engineering Constraints}
\label{sec:constraints}

\subsection{Cost}
The project uses exclusively open-source tools and libraries. No licensing fees are incurred. Development hardware is standard university lab equipment.

\subsection{Performance}
QUIC datagrams are bounded by the QUIC DATAGRAM MTU (typically ~1200\,bytes on standard Ethernet after QUIC framing overhead). Payloads exceeding this limit cannot be sent as a single datagram and must be split at the application layer or sent via a reliable stream instead.

\subsection{Safety}
As a pure software networking library with no hardware interaction, no physical safety considerations apply. Data safety is ensured by mandatory TLS~1.3 encryption on all connections.

\subsection{Regulatory Standards}
QUIC implements TLS~1.3 as specified in RFC~8446 \cite{rfc8446} and the QUIC-TLS integration in RFC~9001 \cite{rfc9001}. The library inherits compliance with these standards through the \texttt{quic-go} dependency.

\section{Success Criteria}
\label{sec:success}

\begin{itemize}
  \item All functional requirements FR-1 through FR-10 are implemented and verified by passing automated tests.
  \item Micro-benchmark throughput for PUB/SUB at 64\,B payload is within 20\% of the theoretical UDP socket baseline on loopback.
  \item A subscriber that undergoes a simulated address migration completes the migration and resumes message reception without application intervention.
  \item 0-RTT reconnection latency is measurably lower than cold-start 1-RTT latency.
\end{itemize}

\section{System-Level Design Description}
\label{sec:sys-design}

QuicMQ is a pure-software system: a Go library that applications import and use directly. There is no broker process, no separate server, and no network daemon. Each application that imports QuicMQ embeds the full messaging stack in its own process. Communication between two applications using QuicMQ consists of one or more QUIC connections established directly between them over UDP.

\section{Major Subsystems}
\label{sec:subsystems}

The library is divided into four layers, described from top to bottom:

\begin{description}
  \item[Socket Layer] Implements the eight public socket types. Each socket type encodes a specific messaging pattern (fan-out, alternating request/reply, etc.) and enforces the corresponding API contract (\texttt{Send}, \texttt{Recv}, \texttt{Listen}, \texttt{Dial}).
  \item[Connection Layer] Manages per-peer connection state, including read and write goroutines, message queuing, and reconnection logic. The \texttt{Conn} type in \texttt{conn.go} wraps a \texttt{net.Conn} and runs a continuous read loop.
  \item[Transport Layer] Abstracts the network transport behind the standard \texttt{net.Conn} / \texttt{net.Listener} interfaces. The QUIC transport adapter (\texttt{transport\_quic.go}) maps QUIC streams to \texttt{net.Conn} and QUIC listeners to \texttt{net.Listener}, so all upper layers are transport-agnostic.
  \item[Wire Layer] Defines the binary framing protocol used to encode multi-frame messages on the wire. The same format is used for both stream and datagram sockets.
\end{description}

\section{Design Choices and Trade-offs}
\label{sec:tradeoffs}

\subsection{QUIC Streams vs.\ TCP for Reliable Patterns}
Using QUIC streams instead of TCP eliminates head-of-line blocking across concurrent message streams on the same connection. The trade-off is increased per-packet overhead (QUIC adds ~20\,bytes of framing over raw UDP vs.\ TCP's 20-byte header) and the added complexity of the QUIC handshake. For workloads with a single sequential stream, TCP would perform equivalently; the benefit emerges when multiple independent streams share one connection.

\subsection{Datagram vs.\ Stream for High-Rate Data}
RFC~9221 datagrams incur no retransmission overhead: a lost packet is simply dropped. For high-rate sensor data where a late packet is worse than a dropped one (e.g., real-time telemetry updating at 1\,kHz), this is preferable to stream-based delivery. The trade-off is that the receiver must tolerate gaps in the data and the payload must fit within the QUIC datagram MTU (~1200\,bytes).

\subsection{Single-Process Broker-Free Model}
Eliminating the broker removes the broker as a bottleneck and as a single point of failure, and reduces end-to-end latency by one network hop. The trade-off is that message durability and replay must be handled by the application, not the messaging layer.

\subsection{Thin Translation Layer}
The QUIC configuration used by QuicMQ is deliberately minimal: the datagram socket types add only \texttt{EnableDatagrams = true} to the base QUIC configuration, and the server adds only \texttt{Allow0RTT = true}. No other fields differ. This keeps the mapping between the QuicMQ API and the underlying \texttt{quic-go} library transparent and easy to maintain.

\section{Software Components}
\label{sec:sw-components}

\subsection{Programming Language}
Go was chosen for its first-class goroutine concurrency model, which maps naturally to the per-connection read/write loop pattern, its strong standard library (TLS, net, sync), and because \texttt{quic-go} — the leading open-source QUIC implementation — is written in Go.

\subsection{QUIC Library: quic-go}
\texttt{quic-go} \cite{quicgo} implements RFC~9000 (QUIC transport), RFC~9001 (QUIC-TLS), and RFC~9221 (QUIC datagrams). It exposes a typed API: \texttt{quic.Listener}, \texttt{quic.Conn}, \texttt{quic.Stream}. QuicMQ wraps these types behind the standard \texttt{net.Listener} / \texttt{net.Conn} interfaces so that socket implementations above the transport layer need no QUIC-specific code.

\subsection{TLS Configuration}
QuicMQ ships two TLS helper functions: \texttt{GenerateTLSConfig()} generates a self-signed certificate for the server, and \texttt{InsecureClientTLSConfig()} returns a client configuration that skips certificate verification, suitable for development and testing. Production deployments should supply a properly signed certificate.

\section{Engineering Calculations}
\label{sec:calc}

\subsection{Wire Protocol Overhead}
Each message frame is prepended with a 5-byte header: 1 flag byte plus a 4-byte big-endian length field. For a 64\,B payload, the overhead ratio is $5 / (5 + 64) \approx 7.2\%$. For a 4\,KiB payload, it falls to $5 / (5 + 4096) \approx 0.12\%$. The format is intentionally compact to avoid dominating small-message throughput.

\subsection{Datagram Size Constraint}
QUIC adds approximately 32\,bytes of header and AEAD tag per datagram (connection ID, packet number, frame header, AEAD tag). On an Ethernet path with MTU 1500\,bytes and a 20-byte IPv4 header plus 8-byte UDP header, the usable QUIC payload is approximately:
\[
  1500 - 20 - 8 - 32 = 1440 \text{ bytes}
\]
After the QuicMQ 5-byte frame header, a single datagram can carry up to 1435\,bytes of application payload. Larger messages must either use a reliable stream or be fragmented at the application layer.

\subsection{Flow-Control Window Sizing}
The QUIC flow-control windows are configured as shown in Table~\ref{tab:windows}. Values are set to exceed the bandwidth-delay product of a 10\,Gbps LAN with 1\,ms RTT ($10 \times 10^9 \times 10^{-3} / 8 = 1.25\,\text{MiB}$), ensuring the window never becomes the bottleneck on local networks.

\section{Related Work}
\label{sec:related}

\subsection{Broker-Based Messaging}
Apache Kafka \cite{kafka}, RabbitMQ, and Apache ActiveMQ centralise message routing in a dedicated broker. Producers publish to named topics; the broker stores and forwards to consumers. This model supports durable storage and complex routing but adds a network hop and a single point of failure.

\subsection{Brokerless Messaging}
ZeroMQ \cite{zmq} pioneered peer-to-peer brokerless messaging. Nanomsg \cite{nanomsg} and NNG refined the transport-agnostic socket abstraction. Both support TCP, in-process, and IPC transports but do not support QUIC.

\subsection{QUIC in Messaging}
NATS \cite{nats} has added experimental QUIC support in its server, but retains a broker architecture. No production-grade brokerless Go library with QUIC transport and a ZeroMQ-compatible API existed in the open-source ecosystem prior to this project.

  \caption{DatagramPub/DatagramSub topology. Subscription commands travel on a reliable QUIC stream (solid line); data payloads travel as RFC~9221 datagrams (dashed line). Both share one UDP flow.}

\end{figure}

Listing~\ref{lst:serial} shows the datagram serialisation function.

\begin{lstlisting}[caption={Datagram serialisation — \texttt{datagram.go}.}, label=lst:serial]
func serializeMsgDatagram(msg Msg) []byte {
    var buf bytes.Buffer
    n := len(msg.Frames)
    for i, frame := range msg.Frames {
        var flag byte
        if i < n-1 {
            flag = 0x01
        }
        buf.WriteByte(flag)
        var lb [4]byte
        binary.BigEndian.PutUint32(lb[:], uint32(len(frame)))
        buf.Write(lb[:])
        buf.Write(frame)
    }
    return buf.Bytes()
}
\end{lstlisting}

The publisher maintains a list of \texttt{dgpeer} structs, each holding a \texttt{quic.Conn} and a topic set. When \texttt{Send} is called, the publisher iterates the peer list, checks topic prefix matching, and calls \texttt{qconn.SendDatagram} for each matching peer.

\section{0-RTT Session Resumption}
\label{sec:0rtt}

The server sets \texttt{Allow0RTT = true}. On first connection the client receives a TLS session ticket and stores it in a \texttt{tls.ClientSessionCache}. On reconnection, \texttt{tr.DialEarly} sends the session ticket in the first packet, allowing application data to be included as 0-RTT early data. QuicMQ detects \texttt{quic.Err0RTTRejected} and transparently falls back to a 1-RTT connection by calling \texttt{qconn.NextConnection}.

\section{Connection Migration}
\label{sec:migration}

QUIC identifies connections by opaque connection IDs, not by source IP/port. When a client's network address changes, \texttt{quic-go} validates the new path using QUIC PATH\_CHALLENGE/PATH\_RESPONSE frames and migrates the connection automatically. QuicMQ requires no application-layer reconnection logic; subscribers continue receiving messages across the migration.

\section{QLOG Tracing}
\label{sec:qlog-impl}

When \texttt{QLOGDIR} is set, QuicMQ injects \texttt{makeQlogTracer(dir)} as the \texttt{quic.Config.Tracer}. This writes one \texttt{.sqlog} file per connection, recording every QUIC packet, frame, and connection event. When \texttt{QLOGDIR} is not set, the default tracer (\texttt{qlog.DefaultConnectionTracer}) is used, which writes nothing unless the environment variable is present, incurring zero overhead on the data path.

\section{Integration Steps}
\label{sec:integration}

A typical integration sequence for a PUB/SUB application is:

\begin{enumerate}
  \item Publisher calls \texttt{NewPub(ctx)} and \texttt{pub.Listen("quic://0.0.0.0:5555")}.
  \item Subscriber calls \texttt{NewSub(ctx)}, \texttt{sub.Dial("quic://server:5555")}, and \texttt{sub.SetOption(OptionSubscribe, "sensor")}.
  \item Publisher calls \texttt{pub.Send(msg)} with the first frame set to the topic (e.g., \texttt{"sensor/temperature"}).
  \item Subscriber calls \texttt{sub.Recv()} in a loop.
\end{enumerate}

For datagram sockets, the same sequence applies with \texttt{NewDatagramPub} / \texttt{NewDatagramSub}; the subscription command is automatically sent over the control stream established during \texttt{Dial}.

\section{Failure Cases and Reasons}
\label{sec:failures}

During development the following failure scenarios were encountered and resolved:

\begin{itemize}
  \item \textbf{0-RTT rejection on server restart:} After a server restart, cached TLS session tickets become invalid. The initial implementation panicked on \texttt{quic.Err0RTTRejected}; the fix adds a \texttt{NextConnection} fallback path.
  \item \textbf{DatagramSub missing messages before subscription propagates:} The publisher fan-out checks topic subscriptions synchronously. A race between \texttt{Dial} completion and the first \texttt{Send} could drop the initial message if the subscription command had not yet been processed. The fix is a brief propagation delay in tests and documentation advising a small wait after \texttt{SetOption}.
  \item \textbf{UDP buffer size warning:} \texttt{quic-go} logs a warning when the OS UDP receive buffer is smaller than recommended (7\,MiB). This is a kernel tunable (\texttt{net.core.rmem\_max}), not a library bug; it does not affect correctness on loopback.
\end{itemize}

\section{Engineering Insights}
\label{sec:insights}

\begin{itemize}
  \item QUIC's stream multiplexing benefit is only observable when multiple independent message streams are active concurrently. For a single sequential PUB/SUB stream, QUIC and TCP perform comparably on loopback.
  \item The datagram socket pair demonstrates a unique capability: reliable control and unreliable data sharing one encrypted UDP socket. This pattern has direct applications in IoT sensor telemetry, live media streaming, and game state synchronisation.
  \item Go's goroutine model maps cleanly to QUIC's per-connection goroutine requirements; the \texttt{quic-go} API design reinforces this.
\end{itemize}

\section{Summary of What Was Built and Tested}
\label{sec:summary}

This project designed and implemented QuicMQ — a brokerless messaging library for Go that uses the QUIC transport protocol (RFC~9000) and its unreliable datagram extension (RFC~9221). Eight socket types were implemented (PUB, SUB, XPUB, XSUB, REQ, REP, DatagramPub, DatagramSub) and validated through unit, integration, scenario, and benchmark tests. All functional requirements FR-1 through FR-10 were implemented; all automated tests pass under the Go race detector.

\section{Whether the Project Met Its Objectives}
\label{sec:objectives-met}

\begin{enumerate}
  \item \textbf{ZeroMQ-compatible API over QUIC streams:} \checkmark\ Implemented. PUB/SUB, REQ/REP, XPUB/XSUB operate over QUIC streams with the same API contract as equivalent ZeroMQ socket types.
  \item \textbf{Datagram socket pair:} \checkmark\ Implemented. DatagramPub/DatagramSub delivers unreliable payloads via RFC~9221 datagrams with subscription control on a reliable stream.
  \item \textbf{Connection migration:} \checkmark\ Transparent via \texttt{quic-go}; scenario tests verify subscriber continuity across address changes.
  \item \textbf{0-RTT resumption:} \checkmark\ Implemented with automatic rejection fallback.
  \item \textbf{QLOG tracing:} \checkmark\ Implemented; \texttt{.sqlog} files are produced when \texttt{QLOGDIR} is set.
  \item \textbf{Benchmarks:} \checkmark\ Infrastructure in place; numeric results depend on target hardware and are reported in Chapter~\ref{ch:results}.
\end{enumerate}

\section{Limitations}
\label{sec:limitations}

\begin{itemize}
  \item All benchmarks are conducted on loopback; wide-area performance under realistic packet loss and jitter has not been measured.
  \item Datagram payloads larger than approximately 1435\,bytes must be split at the application layer; the library does not yet provide automatic fragmentation.
  \item The library does not implement message persistence or durable replay.
  \item TLS certificate management (beyond development self-signed certificates) is left to the application.
\end{itemize}

\section{Suggestions for Improvement}
\label{sec:suggestions}

\begin{itemize}
  \item Implement application-layer datagram fragmentation and reassembly for payloads exceeding the QUIC datagram MTU.
  \item Add a durable pub/sub layer (e.g., write-ahead log) on top of the reliable stream transport.
  \item Automate wide-area benchmarks using a network emulator (e.g., \texttt{tc netem}) to simulate packet loss, reordering, and bandwidth limits.
  \item Provide a connection migration test harness that changes the client's local port programmatically.
\end{itemize}

\section{Opportunities for Commercialisation or Further R\&D}
\label{sec:commercialisation}

\begin{itemize}
  \item \textbf{Industrial IoT:} The datagram socket pair is well-suited for high-rate sensor telemetry in factory automation, where late data is less useful than fresh data.
  \item \textbf{Edge computing / service mesh:} QuicMQ could serve as a QUIC-native sidecar transport in Envoy or Istio, replacing HTTP/gRPC for internal microservice communication.
  \item \textbf{Multi-language SDK:} Exposing the library via a C shared library or gRPC interface would allow Python, Java, and Rust clients to interoperate with Go publishers.
  \item \textbf{Further academic research:} The mixed reliable/unreliable channel pattern (control stream + datagram payload on one QUIC connection) is novel and could be studied as a general design pattern for application protocols.
\end{itemize}
